### 왜 1인 개발자에게도 표준화가 중요한가요?

```
┌─────────────────────────────────────────────────────────────────────┐
│                   1인 개발자의 현실                                  │
└─────────────────────────────────────────────────────────────────────┘

6개월 전에 만든 프로젝트...

오늘의 나: "이 프로젝트 다시 열어볼까"

           Error: Cannot find module 'old-package'

           "어? 환경설정 어떻게 했더라...?"
           "이 함수 왜 만들었지...?"
           "배포는 어떻게 했더라...?"
           
           3시간 후...
           "아 그냥 처음부터 다시 만들까..."

─────────────────────────────────────────────────────────────────────

만약 CLAUDE.md가 있었다면...

오늘의 나: "이 프로젝트 다시 열어볼까"

           프로젝트 실행 완료! ✅

### 1인 개발자를 위한 최소 문서화

팀이 없어도 이것만은 만들어두세요:

```markdown
# CLAUDE.md (1인 개발자 버전)

## 프로젝트 소개
이 프로젝트가 뭐하는 건지 한 줄로.
(6개월 후의 나도 이해할 수 있게!)

## 빠른 시작
```bash
npm install
cp .env.example .env.local
# .env.local에서 API_KEY 설정 필요
npm run dev
```

## 중요한 결정들
| 날짜 | 결정 | 이유 |
|------|------|------|
| 2024-01-15 | React 대신 Next.js | SSR 필요해서 |
| 2024-01-20 | Tailwind CSS 사용 | 빠른 스타일링 |
| 2024-02-01 | Supabase 선택 | 무료 티어 넉넉함 |

## 배포 방법
```bash
vercel
```

## 자주 하는 실수 (내가 했던 것들)
- `.env` 파일 커밋하지 말 것!
- `next.config.js` 수정 후 서버 재시작 필요
- Supabase RLS 정책 확인할 것

## 나중에 할 것들
- [ ] 다크모드 추가
- [ ] 이미지 최적화
- [ ] 성능 개선
```

### 🆕 실습: 개인 프로젝트 표준화하기

**목표:** 6개월 후의 나도 5분 안에 이해할 수 있는 프로젝트 만들기

#### 단계 1: CLAUDE.md 작성 (10분)

```bash
# 프로젝트 폴더에서
claude

> 이 프로젝트의 CLAUDE.md를 만들어줘. 
> 6개월 후에 내가 다시 봐도 이해할 수 있게,
> 환경설정, 중요한 결정, 배포 방법을 포함해서.
```

Claude가 생성한 CLAUDE.md를 검토하고 수정하세요.

#### 단계 2: 자주 쓰는 Commands 만들기 (15분)

```markdown
<!-- .claude/commands/quick-start.md -->
# 프로젝트 빠른 시작

6개월 만에 돌아온 나를 위한 가이드!

## 1. 환경 체크
```bash
node --version  # 20.x 필요
npm --version   # 10.x 이상
```

## 2. 의존성 설치
```bash
npm install
```

## 3. 환경변수 설정
```bash
cp .env.example .env.local
```

`.env.local`에서 필요한 값:
- `DATABASE_URL`: Supabase 연결 문자열
- `NEXT_PUBLIC_SUPABASE_URL`: Supabase 프로젝트 URL
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`: Supabase anon 키

이 값들은 Supabase 대시보드 > Settings > API에서 확인

## 4. 실행
```bash
npm run dev
```

http://localhost:3000 에서 확인!

## 5. 배포
```bash
npm run build && vercel --prod
```
```

#### 단계 3: 결정 기록 습관 만들기

**ADR (Architecture Decision Record)** 패턴을 간단하게 적용:

```markdown
<!-- docs/decisions/001-database-choice.md -->
# Database 선택: Supabase

## 상태
승인됨 (2024-01-20)

## 맥락
- 혼자 개발하는 사이드 프로젝트
- 예산: 월 $0 (무료 티어만)
- 필요 기능: Auth, Database, Storage

## 결정
Supabase를 사용한다.

## 이유
- PostgreSQL 기반 (나중에 이전 쉬움)
- 무료 티어로 충분 (500MB DB, 1GB Storage)
- Auth 내장
- 실시간 구독 지원

## 고려했던 대안
- **Firebase**: NoSQL이라 복잡한 쿼리 어려움
- **PlanetScale**: MySQL이라 덜 익숙
- **직접 DB 운영**: 관리 부담 큼

## 결과
만약 사용자가 많아지면 Supabase Pro ($25/월)로 업그레이드 예정
```

## 왜 팀 협업이 중요한가요?

### 실제 상황: 혼자 vs 팀

**혼자 사용할 때:**
```
나: "이 코드 리뷰해줘"
Claude: (각자의 스타일대로 리뷰)
```

**팀에서 사용할 때:**
```
팀원 A: "/review" → 표준화된 리뷰 프로세스
팀원 B: "/review" → 동일한 리뷰 프로세스
팀원 C: "/review" → 동일한 리뷰 프로세스

결과: 일관된 코드 품질!
```

### 표준화가 없으면 생기는 문제

```
┌─────────────────────────────────────────────────────────────────────┐
│                     표준화 없는 팀의 혼란                             │
└─────────────────────────────────────────────────────────────────────┘

팀원 A: "Claude야, 코드 리뷰해줘"
        → 자기만의 프롬프트 사용
        → 자기만의 스타일로 피드백

팀원 B: "이 코드 괜찮아?"
        → 다른 프롬프트 사용
        → 다른 스타일로 피드백

팀원 C: "@reviewer 리뷰 부탁"
        → 또 다른 프롬프트
        → 또 다른 스타일

새 팀원: "...어떻게 해야 하지? 😰"

결과:
- 일관성 없는 코드 품질
- 새 팀원 온보딩 어려움
- 팀원마다 다른 경험
- 지식 공유 실패
```

### 표준화하면 달라지는 것

```
┌─────────────────────────────────────────────────────────────────────┐
│                     표준화된 팀의 협업                               │
└─────────────────────────────────────────────────────────────────────┘

팀원 A: "/review"
팀원 B: "/review"
팀원 C: "/review"
새 팀원: "/review" (첫날부터 사용 가능!)

모두 동일한:
✓ 리뷰 관점
✓ 피드백 스타일
✓ 품질 기준
✓ 문서화 방식

결과:
- 일관된 코드 품질
- 빠른 온보딩
- 팀 전체 생산성 향상
- 지식 자동 공유
```

## 왜 필요한가요?

**실제 상황**: 팀이 Claude Code를 좋아하지만, 모두 다르게 사용합니다. 한 사람은 특정 명령어를 쓰고, 다른 사람은 다른 설정을 씁니다. 새 팀원들이 혼란스러워합니다. 코드 리뷰가 일관성이 없습니다.

팀을 위해 Claude Code를 표준화하면 모두가 같은 방식으로 일하고, 새 멤버를 온보딩하기 쉬워집니다.

### 쉬운 비유: 팀 플레이북

팀 설정을 스포츠 팀 플레이북처럼 생각해보시기 바랍니다:
- 모두가 같은 플레이를 알고 있습니다 (commands)
- 모두가 같은 용어를 씁니다 (agents)
- 새 선수가 빨리 배울 수 있습니다 (onboarding)

### 또 다른 비유: 맥도날드 매뉴얼

전 세계 맥도날드에서 같은 맛의 빅맥을 먹을 수 있는 이유:
- **표준화된 레시피** = CLAUDE.md (프로젝트 규칙)
- **표준화된 절차** = Commands (작업 프로세스)
- **역할 정의** = Agents (전문가 역할)
- **교육 매뉴얼** = 온보딩 문서

팀의 Claude Code 사용도 이렇게 표준화하면, **누가 사용해도 일관된 결과**를 얻을 수 있습니다!

## 💡 비전공자 팁

> **"표준화"가 왜 중요해요?**
>
> 요리를 예로 들면:
>
> **표준화 없이:**
> - 셰프 A: 소금 한 꼬집
> - 셰프 B: 소금 두 숟가락
> - 셰프 C: 소금... 넣었나?
> → 매번 다른 맛!
>
> **표준화하면:**
> - 모든 셰프: 레시피대로 소금 5g
> → 항상 같은 맛!
>
> 코드도 마찬가지입니다. 팀 전체가 같은 "레시피"를 따르면:
> - 코드 스타일이 일관됨
> - 리뷰 품질이 균일함
> - 새 팀원도 쉽게 적응함

## 왜 팀 협업을 알아야 할까?

혼자 쓸 때와 팀에서 쓸 때는 다릅니다:

- **일관성**: 팀원마다 다른 방식 → 혼란
- **온보딩**: 새 팀원이 바로 시작할 수 있어야 함
- **품질**: 팀 전체의 코드 품질 향상

### 개인 사용 vs 팀 사용 비교

| 측면 | 개인 사용 | 팀 사용 |
|------|----------|--------|
| 설정 | 개인 취향대로 | 팀 표준 필요 |
| Commands | 자기만 알면 됨 | 모두가 사용 가능해야 함 |
| Agents | 자유롭게 | 역할 명확히 정의 |
| 문서화 | 선택사항 | 필수! |
| 온보딩 | 없음 | 매우 중요 |

## 팀 표준화

### 공유해야 할 것들

```
my-project/
├── .claude/
│   ├── settings.json    # 팀 설정
│   ├── commands/        # 팀 Commands
│   │   ├── pr.md
│   │   ├── review.md
│   │   ├── onboarding.md
│   │   └── issue-start.md
│   ├── agents/          # 팀 Agents
│   │   ├── senior-reviewer.md
│   │   ├── junior-helper.md
│   │   └── security-expert.md
│   └── skills/          # 팀 Skills
│       ├── deploy.md
│       └── debug.md
├── CLAUDE.md            # 프로젝트 규칙
├── .github/
│   └── workflows/       # CI/CD (Chapter 26)
└── src/
```

이 폴더들을 Git에 커밋하면 팀 전체가 같은 환경에서 작업합니다.

### 팀 설정 파일 예시

```json
// .claude/settings.json
{
  "project": {
    "name": "Team Project",
    "team": "Frontend Team"
  },
  "preferences": {
    "language": "ko",
    "codeStyle": "team-standard"
  },
  "permissions": {
    "autoApprove": [
      "Read",
      "Glob",
      "Grep",
      "Bash(npm test)",
      "Bash(npm run lint)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(git push --force)"
    ]
  },
  "hooks": {
    "preCommit": "npm run lint && npm test"
  }
}
```

### CLAUDE.md 팀 버전

```markdown
# 프로젝트: My Team App

## 팀 정보
- 팀명: Frontend Team
- 연락처: #frontend-dev (Slack)
- 팀 리더: @team-lead

## 기술 스택
- React 18 + TypeScript
- Tailwind CSS
- Jest + React Testing Library
- GitHub Actions

## 코딩 컨벤션

### 네이밍 규칙
| 대상 | 규칙 | 예시 |
|------|------|------|
| 컴포넌트 | PascalCase | `UserProfile.tsx` |
| 훅 | camelCase, use 접두사 | `useAuth.ts` |
| 파일 | kebab-case | `user-profile.tsx` |
| 테스트 | *.test.tsx | `user-profile.test.tsx` |
| 상수 | SCREAMING_SNAKE_CASE | `MAX_RETRY_COUNT` |

### 코드 스타일
- 함수형 컴포넌트만 사용
- Props는 interface로 정의
- 에러 처리 필수 (try-catch)
- 주석은 한국어로

## Git 규칙

### 브랜치 전략
- `main`: 프로덕션 배포
- `develop`: 개발 통합
- `feature/*`: 기능 개발
- `fix/*`: 버그 수정
- `hotfix/*`: 긴급 수정

### 커밋 메시지
Conventional Commits 사용:
- `feat:` 새 기능
- `fix:` 버그 수정
- `docs:` 문서
- `refactor:` 리팩토링
- `test:` 테스트
- `chore:` 기타

예시: `feat(auth): Add social login support`

### PR 규칙
- PR 필수: main 브랜치 직접 푸시 금지
- 리뷰어: 최소 1명
- CI 통과 필수
